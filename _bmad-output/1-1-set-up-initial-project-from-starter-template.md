## Dev Notes

### Epic Context

- Epic 1 objective: enable scenario execution with clear, traceable results. Story 1.1 is the foundation slice that unblocks all subsequent stories in Epic 1.
- Downstream dependencies:
  - Story 1.2 assumes usable `apps/web` + `apps/api` boundaries and runnable scripts.
  - Story 1.3 assumes contract/persistence package boundaries are in place.

### Technical Requirements

- Locked stack must be preserved from day zero:
  - Frontend: Next.js App Router + Tailwind + shadcn/ui + TanStack Query + React Hook Form.
  - Backend: NestJS.
  - Data: PostgreSQL + Prisma (schema/migrations later stories).
  - Observability: Prometheus + Loki + Grafana + Sentry (infra scaffolding can start here, full wiring later).
- Architecture-selected starter approach is mandatory:
  - `create-turbo` monorepo baseline.
  - `apps/web` initialized with Next.js TypeScript App Router.
  - `apps/api` initialized with Nest CLI TypeScript.
- Keep implementation deterministic and reviewer-friendly: no hidden manual setup steps.

### Architecture Compliance Guardrails

- Preserve strict boundaries:
  - UI work in `apps/web`
  - API orchestration in `apps/api`
  - Shared contracts only in `packages/contracts`
- Keep frontend thin; do not introduce direct observability/database calls in UI.
- Maintain naming conventions:
  - files: `kebab-case`
  - classes/components: `PascalCase`
  - variables/functions: `camelCase`
- Keep docs/checklist as first-class deliverables; onboarding docs are part of story completeness.

### Library/Framework Requirements

- Use current stable major lines from architecture:
  - Next.js 16.x
  - NestJS 11.x
  - TanStack Query 5.x
  - React Hook Form 7.x
  - Prisma 7.x
- Web cross-check (non-authoritative but current) indicates:
  - Next.js 16.2.x and Prisma 7.7.x available
  - NestJS 11.1.x available
- Prefer exact versions generated by official CLIs, then pin/normalize in workspace.

### File Structure Requirements

- Minimum required structure after this story:
  - `apps/web`
  - `apps/api`
  - `packages/contracts`
  - root workspace/tooling files
  - root `README.md`
  - root `.env.example`
- Future-required structure should be seeded (empty placeholders acceptable if not implemented yet):
  - `infra/observability/*` paths
  - app-level env examples (`apps/web/.env.local.example`, `apps/api/.env.example`)

### Testing Requirements

- For this story, baseline verification is command-level and structural:
  - dependency installation succeeds
  - workspace scripts execute without structural/tooling errors
  - app boundaries exist as expected
- Do not claim done unless baseline checks are actually run and outcomes are recorded.
- Keep verification deterministic; avoid flaky/time-based assertions in early setup scripts.

### Implementation Notes for Dev Agent

- Use official scaffolding CLIs first, then normalize project scripts/config.
- Prefer minimal modifications to generated files; focus on structural correctness.
- Avoid introducing product logic in this story (no scenario orchestration implementation yet).
- Record exact commands used and outputs in completion notes.
- If tool versions conflict, resolve with minimal, documented adjustments and keep stack locked.

### Project Structure Notes

- This repository currently contains docs and BMAD outputs only; no application scaffold exists yet.
- Story 1.1 is responsible for creating first runnable project skeleton.
