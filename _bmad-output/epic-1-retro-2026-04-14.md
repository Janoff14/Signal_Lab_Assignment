# Epic 1 Retrospective - 2026-04-14

## Epic Review

### Scope Reviewed
- Epic 1: Run scenarios with clear, traceable results.
- Stories completed: `1-1` through `1-6`.

### What Went Well
- Core vertical slice was implemented end-to-end early (`POST /api/v1/scenario-runs` + UI form flow).
- Shared contracts in `packages/contracts` kept web/api integration stable.
- Validation gates (`test:structure`, `lint`, `typecheck`, `test`, `build`) were run consistently.
- Story tracking hygiene improved over time with complete status and file-list updates.

### What Did Not Go Well
- Initial workflow drift: later stories were marked complete before a separate explicit code-review pass per story.
- First-pass implementation introduced a few contract/robustness gaps later found in code review (error envelope for non-HTTP errors, signal query validation, run ID collision risk).
- Story docs became inconsistent in structure and needed normalization.

### Key Lessons
- Keep submission-ack semantics explicit (`running`) and avoid blending submission with final completion semantics.
- Preserve strict API envelope behavior for both expected and unexpected errors.
- Use explicit DTO validation for query inputs, not only request bodies.
- Keep story document format uniform from the beginning to reduce post-pass cleanup.
